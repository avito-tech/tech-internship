`README` — первое, что видит проверяющий. Если он оформлен понятно и подробно, разобраться в вашем проекте будет гораздо легче.

**Что стоит включить в README:**

- Инструкцию по запуску приложения.
- Скриншоты или запись экрана, чтобы продемонстрировать работу приложения.
- Если возникали вопросы при разработке, и вы их решали — можно вынести в отдельный блок с пояснением решений.

## Обработка ошибок и разрешений

Не забывайте делать обработку ошибок при выполнении работы. 
Важно, чтобы пользователь понимал, что происходит, а приложение продолжало работать. Также обратите внимание на работу с системными разрешениями.

**Основные принципы работы с разрешениями:**
1. Запрашивайте permission только тогда, когда он необходим для работы приложения.
2. Показывайте пользователю обоснование запроса до запроса разрешения.
3. Обрабатывайте отказ и блокировку разрешения.
4. Обновляйте UI после результата разрешения.

## Чистая архитектура

Хорошая архитектура повышает поддерживаемость, надёжность и масштабируемость приложения, а также облегчает тестирование.

**Рекомендации по архитектуре Android-приложений:**
- *Разделение на слои:*
    - UI-слой — отображение UI и взаимодействие с пользователем;
    - Data-слой — получение и хранение данных, бизнес-логика;
    - Domain-слой — полезен для переиспользования бизнес-логики.
- ViewModel должна использовать репозитории, а не напрямую делать сетевые вызовы или обращаться к базе.
- Для асинхронных операций рекомендуется использовать Coroutines и Flow.
- *Unidirectional Data Flow (UDF):* ViewModel публикует состояние, UI подписывается на него, а события UI отправляются обратно в ViewModel.
- ViewModel не должна зависеть от Android-фреймворка, чтобы её можно было тестировать независимо от платформы.

## Многомодульность

Важно продумывать логическое разделение проекта на модули.

**Преимущества многомодульной архитектуры:**
1. *Низкая связность:* модули не зависят друг от друга напрямую.
2. *Инкапсуляция:* наружу доступен только публичный API, детали реализации скрыты.
3. *Переиспользование кода:* общие компоненты удобно хранить в `core/common` и подключать в разных фичах.
4. *Быстрая сборка:* Gradle пересобирает только изменённые модули.
5. *Облегчение тестирования:* каждый модуль можно тестировать отдельно.

**Советы по модульной структуре:**
- Минимизируйте публичный API модуля, скрывайте детали через `internal` или `private`.
- Используйте `implementation` вместо `api`, чтобы зависимости модуля не прорастали дальше и сборка шла быстрее.
- Удаляйте неиспользуемые библиотеки и зависимости, чтобы не усложнять проект.
