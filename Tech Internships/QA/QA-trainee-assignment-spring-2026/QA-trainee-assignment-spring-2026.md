# Задание 2.1: Тесты API

**Условие:**

*Мы имеем микросервис, содержащий 4 ручки.*

* Создать объявления  
* Получить объявления по его идентификатору  
* Получить все объявления по идентификатору продавца  
* Получить статистику по айтем id  
* Host — https://qa-internship.avito.com

**Postman-коллекцию можно найти здесь.**

**Особенности:**

* Каждое созданное объявление получает уникальный идентификатор.  
* Поля name, price, sellerId, statistics могут повторяться у разных объявлений.  
* sellerId — целое число. Рекомендуем использовать значения в диапазоне 111111–999999, чтобы избежать пересечений с другими тестировщиками.

**Суть задания:**

1. Составьте тест-кейсы для проверки API. Опишите их в файле TESTCASES.md.  
   Тест-кейсы должны покрывать:

* Позитивные сценарии  
* Негативные сценарии  
* Корнер-кейсы. Например, идемпотентность запросов  
* Функциональные и нефункциональные проверки  
* Другие важные аспекты, которые вы считаете нужными.  
  *Также не забывайте использовать техники тест-дизайна.*  
2. Автоматизируйте написанные тест-кейсы.

* Реализуйте проверки с помощью assert.  
* Тесты должны быть независимыми и воспроизводимыми.  
* Если нужно, генерируйте уникальные sellerId в диапазоне 111111–999999.  
* Все тесты должны успешно проходить (если сервис работает корректно) или падать с обоснованием в баг-репортах.  
3. Оформите результаты:  
   * TESTCASES.md — полное описание тест-кейсов.  
   * README.md — инструкция по запуску автоматических тестов (установка зависимостей, настройка, команда запуска).  
   * BUGS.md — если вы обнаружите дефекты, опишите их по шаблону: краткое описание, шаги воспроизведения, фактический результат, ожидаемый результат, серьёзность, окружение.

**Дополнительные задания (необязательные, но дают преимущество)**

Выполнение одного или нескольких пунктов ниже позволит продемонстрировать более глубокие навыки тестирования и автоматизации.

1. Тестирование с моками для проверки обработки ошибок (ошибки сервера, таймауты, некорректные ответы).

Проверить сценарии:

* сервер возвращает 500 при создании объявления  
* сервер возвращает 404 для существующего ID (симуляция временной недоступности)  
* таймаут соединения  
* ответ с некорректным JSON (например, битые данные)  
* пустой список объявлений продавца вместо ожидаемого массива  
* и другие ситуации на ваше усмотрение  
* Убедитесь, что ваши проверки корректно обрабатывают такие случаи (например, выдают осмысленное сообщение об ошибке, а не падают с исключением).

2. E2E-тестирование

Автоматизируйте кейсы, объединяющие несколько шагов.

3. Формирование отчётов с использованием Allure

Задачи:

* Подключить Allure  
* Добавить в тесты шаги, заголовки, описания и при необходимости вложения (скриншоты, логи)  
* Запустить тесты с сохранением результатов в формате Allure, сгенерировать отчёт и открыть его локально  
* Приложить скриншоты или пример отчёта в решении (можно сохранить как allure-example.html).

4. Настройка линтера и форматтера для тестового кода

Задачи:

* Создать конфигурационные файлы и зафиксировать в них правила.  
* Описать конфигурацию и команды для запуска в README.md.

# Задание 2.2: Тесты UI

**Условие**  
   
На [сайте](https://cerulean-praline-8e5aa6.netlify.app/) размещена платформа модерации объявлений.  
Платформа состоит из:

* главной страницы, где находят объявления (`/`, `/list`)  
* страницы карточки объявления, где происходит сама  модерация (`/item/:*id*`)  
* страницы просмотра статистики (`/stats`)

   
**Суть задания:**  
 

1. Составьте тест-кейсы на пользовательские сценарии из списка ниже:  
   1. На десктопной версии сайта:  
      1. **Главная** страница: Проверить работу фильтра «**Диапазон цен**»  
      2. **Главная** страница: Проверить сортировку «**По цене**»  
      3. **Главная** страница: Проверить работу фильтра «**Категория»**  
      4. **Главная** страница: Проверить работу тоггла «**Только срочные**»  
      5. Страница **Статистики**: Протестировать контейнер управления **таймером обновления статистики.** (Работает кнопка «Обновить». Работает кнопка остановки таймера. Работает кнопка запуска таймера).  
   2. На мобильной версии сайта:  
      1. Проверить переключение между **тёмной и светлой темой.**  
2. Автоматизируйте тест-кейсы:  
   1. Обязательна проверка результатов (assertions).  
   2. При обнаружении бага отчёт после прогона должен явно это отражать.  
   3. Отчёт должен быть лаконичным и понятным.  
3. Запустите автоматизированные тесты:  
   1. Все тесты проходят, если багов нет.  
   2. Не допускаются «сломанные» тесты: тест либо падает из-за найденного бага, либо проходит успешно.

   
**Что стоит учитывать:**  
   
Обратите внимание: вы работаете с общим тестовым стендом.  
Приложение не изолированно для каждого кандидата, поэтому данные могут изменяться другими пользователями во время выполнения задания. Учитывайте это при написании автотестов — тесты должны быть устойчивыми к изменяющемуся состоянию системы. При разработке автотестов учитывайте, что начальное состояние данных не гарантировано и может меняться в любой момент.  
Поэтому:

* не стоит опираться на фиксированное количество элементов  
* не следует предполагать неизменность конкретных записей.

   
**Как повысить свои шансы на высокую оценку:**  
   
В README.md нужно максимально лаконично и понятно описать, как запустить тесты. Проект должен быть легко воспроизводим: после клонирования репозитория тесты должны запускаться стандартной командой без длительной ручной настройки окружения.  
   
**Нефункциональные требования:**

* Оформите решение с тест-кейсами в файле TESTCASES.md  
* Задание по автоматизации можно выполнить на любом удобном вам фреймворке, драйвере, языке программирования (в автоматизации тестирования это чаще всего JS, TS, Python, Java и т. д.)  
* Оформите инструкцию по запуску в файле README.md  
* Если в результате тестирования найдены баги, то составьте баг-репорт в файле BUGS.md.
