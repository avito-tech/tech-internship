# **Тестовое задание для стажёра Fullstack (единичный найм, 2026) - каталог AI-ассистентов**

## Бизнес-контекст

Внутри компании появляется всё больше прикладных AI-ассистентов: для рецептов, тренировок, код-ревью, составления писем, анализа документов и других задач. Каждый ассистент решает узкую проблему в своей предметной области. Ассистент состоит из:

- системного промпта, который описывает правила поведения ассистента;
- модели, которую нужно использовать для генерации ответа;
- категории, к которой ассистент относится;
- контекста от пользователя, который ассистент ожидает при запуске.

Например, ассистент **«Повар»** относится к категории **«Еда»**. В системном промпте описано, что он должен составлять подробный рецепт: учитывать список ингредиентов, предлагать шаги приготовления, время, сложность и советы по замене продуктов. Пользователь при запуске передаёт только список ингредиентов, а ассистент возвращает рецепт.

Нужно разработать каталог AI-ассистентов в формате веб-приложения, где администратор управляет ассистентами и их категориями, а пользователь просматривает каталог, фильтрует ассистентов по категориям и запускает подходящего ассистента с контекстом, переданным пользователем.

## Описание задачи

Необходимо разработать fullstack-приложение:

- backend-сервис с REST API, JWT-авторизацией и хранением данных в реляционной БД;
- frontend SPA-приложение на React с несколькими страницами, формами, фильтрами, загрузочными и ошибочными состояниями;
- интеграцию frontend и backend через API из файла [api.yaml](api.yaml);
- может использоваться мок LLM-модуля в качестве альтернативы реальному провайдеру (без необходимости использовать реальные, в том числе платные LLM-сервисы).

## Технические требования к решению

- Backend: рекомендуется Go, но допустимы другие языки и платформы: Java, PHP, Node.js и т.п.
- База данных: рекомендуется PostgreSQL, но допустимо использовать MySQL, SQLite или любую другую реляционную БД.
- Frontend: React SPA.
- Проект должен запускаться через Docker Compose.
- Файл `docker-compose.yaml` должен находиться в корне репозитория кандидата.
- Команда запуска:
```bash
docker-compose up --build
```

- Все переменные окружения в `docker-compose.yaml` должны иметь дефолтные значения.
- Backend должен слушать порт `8080`.
- Frontend должен быть доступен на порту, указанном в `README.md`.
- `GET /_info` должен возвращать `200 OK`.
- Использовать реальное LLM API не обязательно, можно обойтись моком LLM внутри backend-сервиса.

## Роли пользователей

**Администратор** может:

- создавать категории ассистентов;
- создавать и редактировать ассистентов;
- просматривать список всех запусков ассистентов;
- включать и выключать ассистентов.

**Пользователь** может:

- просматривать список категорий;
- просматривать список ассистентов;
- фильтровать ассистентов по категории;
- искать ассистентов по названию и описанию;
- открывать карточку ассистента;
- запускать ассистента с учётом пользовательского контекста;
- просматривать историю своих запусков.

## Предметная область

**Пользователь** — участник системы с уникальным идентификатором, email, ролью (`admin` / `user`) и датой регистрации.

**Категория** — предметная область, к которой относится ассистент. Например: `Еда`, `Спорт`, `Программирование`, `Путешествия`.

**AI-ассистент** — сущность каталога. Содержит название, описание, `categoryId`, модель, системный промпт, пример пользовательского промпта, признак активности и дату создания.

Пример ассистента:

| Название | Категория | Модель | Контекст пользователя | Системный промпт | Ожидаемый результат |
| --- | --- | --- | --- | --- | --- |
| Повар | Еда | gpt-4o-mini | `курица, лаваш, огурцы, томаты, соус` | Составь простой домашний рецепт из ингредиентов пользователя: дай название, примерное время и 3-5 коротких шагов. | Домашняя шаверма за 20 минут: нарежь курицу полосками и обжарь до золотистой корочки, огурцы и томаты порежь тонкими ломтиками, смажь лаваш соусом, выложи овощи и курицу, плотно заверни и подрумянь на сухой сковороде по минуте с каждой стороны. |


**Модель** — строковый идентификатор модели, которую ассистент должен использовать. Например, `gpt-4o-mini`.

**Системный промпт** — инструкция, привязанная к ассистенту. Пользователь не редактирует её при запуске, но backend использует её при формировании запроса к LLM-провайдеру.

**Пользовательский промпт** — входные данные, которые пользователь передаёт ассистенту при запуске. Например, для ассистента `Повар`: `курица, рис, томаты, сыр`.

**Запуск ассистента (AssistantRun)** — сохранённый факт обращения пользователя к ассистенту: `assistantId`, `userId`, пользовательский промпт, итоговый ответ, статус, ошибка, дата создания.

Статусная модель запуска:

- `pending` — запуск создан, но ответ LLM-провайдера ещё не получен. Для синхронной реализации этот статус может быть короткоживущим, но модель данных и API должны его поддерживать;
- `success` — запуск успешно завершён, поле `output` содержит ответ ассистента;
- `failed` — запуск завершился ошибкой, поле `error` содержит понятное описание ошибки.

## Backend: обязательные требования

### API

Реализация должна соответствовать спецификации [api.yaml](api.yaml).

Обязательные эндпоинты:

- `POST /dummyLogin` — выдаёт тестовый JWT по роли `admin` или `user`;
- `GET /categories` — список категорий;
- `POST /categories` — создание категории, только для роли `admin`;
- `GET /assistants` — список ассистентов с фильтрацией по категории, поиском по названию и описанию и пагинацией;
- `POST /assistants` — создание ассистента, только для роли `admin`;
- `GET /assistants/{assistantId}` — карточка ассистента;
- `PUT /assistants/{assistantId}` — редактирование ассистента, только для роли `admin`;
- `POST /assistants/{assistantId}/run` — запуск ассистента пользователем;
- `GET /runs/my` — история запусков текущего пользователя;
- `GET /admin/runs` — список всех запусков, только для роли `admin`;
- `GET /_info` — healthcheck, всегда возвращает статус `200`.

### Авторизация

- Для авторизации используется JWT.
- Тестовый токен выдаётся через `/dummyLogin`.
- JWT должен содержать как минимум `user_id` и `role`.
- Backend должен проверять права доступа на уровне middleware или другого централизованного механизма.
- При запуске ассистента `user_id` берётся из токена, а не из тела запроса.

`/dummyLogin` должен возвращать фиксированный UUID для каждой роли:

- один и тот же UUID для всех запросов с ролью `admin`;
- один и тот же UUID для всех запросов с ролью `user`.

Это нужно, чтобы проверяющий мог стабильно воспроизводить сценарии работы с историей запусков.

### База данных

- Рекомендуется использовать PostgreSQL, но допустимо использовать другую реляционную БД.
- Схема БД должна описывать пользователей, категории, ассистентов и запуски ассистентов.
- Миграции должны применяться автоматически при запуске приложения или отдельной документированной командой.
- Нужно продумать индексы для списка ассистентов с фильтрацией по категории и поиска по названию.
- Состояние приложения должно сохраняться между перезапусками контейнера, если volume не удалён.

### Бизнес-ограничения

- Ассистент всегда должен быть привязан к существующей категории по `categoryId`.
- Нельзя создать ассистента без системного промпта.
- Нельзя запустить неактивного ассистента.
- Пользователь без роли администратора не может создавать и редактировать категории или ассистентов.
- Список ассистентов должен поддерживать фильтрацию по `categoryId`.
- По умолчанию `GET /assistants` возвращает только активных ассистентов.
- Для администратора должен быть параметр `includeInactive=true`, позволяющий увидеть выключенных ассистентов.
- Каждый запуск ассистента должен сохраняться в БД независимо от успеха или ошибки LLM-провайдера.
- Новый запуск должен создаваться в статусе `pending`, после получения ответа переводиться в `success`, а при ошибке, таймауте или невалидном ответе провайдера — в `failed`.
- Если мокируемый LLM-провайдер вернул ошибку, запуск сохраняется со статусом `failed`, а API возвращает понятную ошибку.
- Ответ ассистента должен формироваться на backend: frontend отправляет только пользовательский промпт и не передаёт системный промпт.

### Запросы к LLM

Реальный внешний LLM API не обязателен и не должен требоваться для запуска проекта. Как уже сказано выше, можно сделать мок LLM-провайдера.

Требования для реализации мока:

- реализовать мок-провайдер, который возвращает детерминированный ответ;
- при запуске ассистента backend формирует запрос из системного промпта ассистента, модели и пользовательского промпта;
- сохранить пользовательский промпт, итоговый ответ, модель, статус и ошибку запуска в БД;
- описать в `README.md` пример итогового запроса, который был бы отправлен в LLM;
- описать обработку таймаута, невалидного ответа и ошибки провайдера.

Дополнительно можно реализовать OpenAI-совместимый провайдер за фича-флагом или переменной окружения, но приложение должно запускаться и проходить проверку без реального ключа.

### Производительность и объёмы данных

Ориентиры:

- до 100 категорий;
- до 10 000 ассистентов;
- до 10 000 пользователей;
- до 1 000 000 запусков ассистентов;
- 100 RPS;
- SLI 99.9%;

## Frontend: обязательные требования

### Стек

- React 18+;
- SPA;
- любая библиотека для роутинга, например `react-router-dom`;
- любой сборщик: Vite, Webpack или аналог;
- любой инструмент для HTTP-запросов: `fetch`, Axios, TanStack Query и т.п.;
- любая библиотека UI-компонентов.

### Страницы

Приложение должно состоять минимум из следующих страниц:

1. `/login` — тестовый вход.
   - Выбор роли: `admin` или `user`.
   - Вызов `/dummyLogin`.
   - Сохранение токена и роли в состоянии приложения.
   - Переход на доступную роли страницу после входа.

2. `/assistants` — каталог ассистентов.
   - Список карточек или таблица ассистентов.
   - Фильтрация по категории.
   - Поиск по названию и описанию.
   - Постраничная пагинация.
   - Для администратора — фильтр `Показать неактивных ассистентов` (в API передаётся через параметр `includeInactive`).
   - Отображение loading/error/empty состояний.

3. `/assistants/:id` — карточка ассистента и запуск.
   - Название, описание, категория, модель, пример пользовательского контекста.
   - Форма для ввода пользовательского контекста.
   - Кнопка запуска ассистента.
   - Отображение ответа.
   - Отображение ошибки, если запуск завершился неуспешно.

4. `/runs/my` — история моих запусков.
   - Список запусков текущего пользователя.
   - Постраничная пагинация.
   - Название ассистента, статус, дата, пользовательский промпт, краткий ответ.
   - Фильтр по статусу.
   - Возможность открыть полный ответ.

5. `/admin/categories/new` — создание категории, только для `admin`.
   - Форма с валидацией.
   - Название и описание категории.

6. `/admin/assistants/new` — создание ассистента, только для `admin`.
   - Форма с валидацией.
   - Название, описание, категория, модель, системный промпт, пример пользовательского промпта, вкл/выкл.

7. `/admin/assistants/:id/edit` — редактирование ассистента, только для `admin`.
   - Редактирование полей ассистента с валидацией.
   - Возможность включить или выключить ассистента.

8. `/admin/runs` — все запуски ассистентов, только для `admin`.
   - Таблица запусков.
   - Постраничная пагинация.
   - Фильтр по ассистенту.
   - Фильтр по статусу.

### Детали реализации

Обязательно:

- вынести работу с API в отдельный слой;
- реализовать централизованное хранение токена и текущей роли;
- защитить роуты по роли;
- отображать состояния загрузки и ошибки на каждой странице с запросами;
- после создания или редактирования ассистента обновлять данные в каталоге;
- после запуска ассистента обновлять историю запусков;
- не показывать системный промпт обычному пользователю, если вы считаете его внутренней настройкой ассистента. Обосновать решение нужно в `README.md`.

Можно использовать Context, Zustand, Redux, MobX, TanStack Query или другой подход. Выбор нужно кратко обосновать в `README.md`.

## Тесты

Код должен быть покрыт тестами:

- backend: юнит-тесты на ключевые бизнес-сценарии;
- backend: интеграционный или E2E-тест на сценарий `admin создал категорию -> admin создал ассистента -> user запустил ассистента`;
- backend: тест, что фильтрация ассистентов по `categoryId` возвращает только ассистентов выбранной категории;
- backend: тест, что запуск неактивного ассистента запрещён;
- frontend: минимум 3 теста на компоненты или пользовательские сценарии;
- Проанализирован coverage и описан в `README.md`.

## Рекомендации по решению

Этот раздел не добавляет новых обязательных требований, но помогает понять, на что будет обращать внимание ревьюер.

### Code style и структура кода

- Используйте автоформатирование и линтеры: `gofmt`/`goimports`, `golangci-lint`, ESLint и Prettier.
- Не складывайте все HTTP-обработчики в один файл. Лучше разделить их по доменам: `auth`, `categories`, `assistants`, `runs`.
- Не смешивайте DTO, модели БД и доменные модели без необходимости. Именованные структуры проще валидировать, тестировать и переиспользовать.
- Моки LLM-провайдера и репозиториев лучше хранить отдельно от production-кода: например, в `mocks/` или `internal/.../mock`.

### Архитектура и разделение слоёв

- HTTP-слой должен заниматься парсингом запроса, валидацией входных данных и маппингом ошибок в HTTP-ответы.
- Бизнес-логику лучше вынести в слой usecase/service: создание ассистента, проверку категории, запуск ассистента, сохранение истории.
- Репозитории должны инкапсулировать работу с БД, а не протекать SQL-запросами в обработчики запросов.
- Интерфейсы зависимостей лучше объявлять на стороне потребителя. Например, usecase запуска ассистента зависит от интерфейса `LLMProvider`, а не от конкретной реализации `MockProvider`.
- Для операции запуска ассистента продумайте порядок действий: загрузка ассистента, проверка активности, вызов LLM-провайдера, сохранение успешного или неуспешного запуска.

### База данных и миграции

- Не ограничивайтесь `init.sql`. Лучше использовать отдельные миграции: `0001_create_categories.sql`, `0002_create_assistants.sql`, `0003_create_runs.sql`.
- Добавьте индексы под основные запросы: фильтрацию ассистентов по `category_id`, поиск по названию/описанию, историю запусков по `user_id` и сортировку по `created_at`.
- Внешние ключи должны защищать целостность данных: ассистент не должен ссылаться на несуществующую категорию.
- Если при запуске ассистента вы сохраняете и запрос, и результат, продумайте транзакционность и поведение при ошибке LLM-провайдера.

### Тесты

- Юнит-тесты должны проверять бизнес-логику, а не только HTTP-коды.
- Для интеграционных и E2E-тестов используйте отдельную тестовую базу данных, чтобы не портить "реальные" данные.
- Хорошая практика — отдельный `docker-compose.e2e.yaml` или отдельный набор переменных окружения для тестов.
- Проверьте негативные сценарии: запуск выключенного ассистента, создание ассистента с несуществующей категорией, доступ обычного пользователя к админским эндпоинтам, ошибка LLM-провайдера.
- Следите, чтобы тесты были стабильными: не завязывайтесь на случайный порядок map, текущее время без фиксации clock и общую БД между тестами.

### Docker, окружение и конфигурация

- Регулярно проверяйте чистый старт через `docker-compose up --build`.
- Добавьте healthcheck для БД и backend, а миграции запускайте предсказуемо.
- Если вы выбрали Go, версии Go в `go.mod` и Dockerfile должны совпадать.
- Используйте multi-stage Dockerfile для backend и frontend.
- Все обязательные переменные окружения должны иметь дефолтные значения в `docker-compose.yaml` или быть описаны в `.env.example`.

### Авторизация и безопасность

- Передавайте JWT стандартно: `Authorization: Bearer <token>`.
- Не коммитьте `.env`, приватные ключи, реальные API-ключи и пароли. В репозитории достаточно `.env.example`.
- Даже если реальная регистрация является дополнительным заданием, пароли нельзя хранить в открытом виде. Используйте bcrypt или аналог.
- Обычному пользователю не обязательно показывать системный промпт ассистента. Если вы скрываете его, убедитесь, что frontend не получает его через API.
- Реальный OpenAI ключ, если вы добавляете реальный провайдер, должен читаться только из переменных окружения.

### LLM-интеграция

- Мок-провайдер должен быть детерминированным, чтобы тесты не зависели от случайного текста.
- Системный промпт должен храниться и применяться на backend. Frontend отправляет только пользовательский промпт.
- Обработайте таймауты, ошибки провайдера и пустые ответы. Неуспешный запуск тоже должен сохраняться в истории со статусом `failed`.

### Frontend

- Вынесите работу с API в отдельный слой, чтобы компоненты не собирали URL и заголовки вручную.
- Защитите маршруты по роли: пользователь не должен попадать на admin-страницы даже при ручном вводе URL.
- Для каталога ассистентов синхронизируйте состояние фильтров с query-параметрами или явно обоснуйте другой подход.
- Отображайте loading, error и пустое состояние на страницах каталога, карточки ассистента и истории запусков.
- Не храните JWT в нескольких несогласованных местах. Выберите единый подход и опишите его в README.

## Docker и запуск

Проект должен запускаться через Docker Compose.

В корне репозитория должны быть:

- `docker-compose.yaml`;
- `README.md`;
- backend-приложение;
- frontend-приложение.

Команда запуска:

```bash
docker-compose up --build
```

После запуска:

- backend доступен на `http://localhost:8080`;
- frontend доступен на `http://localhost:3000` или другом явно указанном в README порту;
- `GET http://localhost:8080/_info` возвращает `200`.

Все переменные окружения в `docker-compose.yaml` должны иметь дефолтные значения, чтобы проект запускался без ручной настройки.
